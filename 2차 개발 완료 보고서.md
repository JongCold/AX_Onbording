# 2차 개발 완료 보고서 (트러블슈팅 및 기능 고도화)

본 보고서는 온보딩 자동화 대시보드 웹 구동 및 RAG 챗봇 시스템 연동 테스트 중 발생한 주요 예외 상황(Troubleshooting)과 이에 대한 조치 방법, 그리고 고도화된 파일 버전 관리 체계를 세부적으로 기술합니다.

## 1. 슬랙 봇 채널 비참여 에러 (`not_in_channel`) 트러블슈팅

### 1.1 현상 및 에러 로그
관리자 대시보드에서 기존 슬랙 채널(예: `#새-채널`)을 선택해 사원 등록 및 PDF 전송을 시작했을 때 아래와 같은 오류가 발생하며 파일 전송과 사용자 초대가 중단됨:
```
Slack user lookup/invite skipped or failed: {'ok': False, 'error': 'not_in_channel'}
Error uploading file to slack: {'ok': False, 'error': 'not_in_channel'}
```

### 1.2 원인 분석
- 슬랙 API 규격상, 봇 토큰(`SLACK_BOT_TOKEN`)은 자신이 참여하고 있지 않은 채널 내부로 다른 멤버를 초대(`conversations.invite`)하거나 파일 업로드(`files.upload_v2`)를 시도할 수 없으며, 이 경우 `not_in_channel` 예외를 반환합니다.
- 봇의 자동 채널 조인을 가능케 하는 `channels:join` 스코프가 봇 앱 권한에 누락되어 강제 조인이 불가했고, 대표자(관리자)의 유저 권한 토큰(`SLACK_USER_TOKEN`) 또한 지정되어 있지 않아 봇 스스로 방에 입장할 통로가 전무했습니다.

### 1.3 해결 조치 및 고도화
1. **유저 권한 토큰을 통한 봇 초대 2중 방어선 구현**:
   - `slack_service.py` 내부 `ensure_bot_in_channel` 메서드에 유저 권한(`SLACK_USER_TOKEN`) 구원 투수 로직을 적용했습니다. 봇 조인 실패 시, 대표 계정의 권한을 대행해 채널에 가입하고 봇(`@Auto_Bot1`)을 강제 초대하여 권한 오류를 자동 치유합니다.
2. **동적 신규 채널 생성 분기 탑재 (`__NEW_CHANNEL__`)**:
   - 웹 대시보드 드롭다운 최상단에 **`🆕 [새로운 전용 채널 자동 생성하여 진행]`** 옵션을 신설하였습니다.
   - 백엔드 수신부([main.py](file:///c:/auto_onbording/main.py))에서 이 옵션 감지 시, 가입자 이메일 접두사를 토대로 전용 온보딩 채널(`ot-proj-{prefix}`)을 동적으로 자동 생성합니다.
   - 슬랙 특성상 **채널을 최초 생성한 생성자(Creator)는 채널에 자동으로 가입**되므로, 별도의 조인 스코프나 토큰 없이도 `not_in_channel` 에러를 원천적으로 우회하고 파일 배포를 완벽히 완료합니다.
3. **가입자 초대 DB 중복 쓰기 버그 제거**:
   - 가입 안내 이메일 발송 과정에서 로컬의 상태 파일(`pending_workers.json`) 쓰기 충돌로 인해 발생했던 UI 정합성 매핑 에러를 제거하고 예외 핸들러를 깔끔하게 다듬었습니다.

## 2. Ollama 로컬 임베딩 실패 및 RAG 지식 무반응 트러블슈팅

### 2.1 현상 및 에러 로그
문서 업로드 및 인덱싱 과정에서 HTTP 200 응답은 오지만 실제 임베딩 벡터가 산출되지 않아 RAG 지식베이스에 정상 머징되지 못하는 현상:
```
[WARNING] Ollama 임베딩 반환 실패 (HTTP 200), 디폴트 제로 벡터 할당.
```
이로 인해 사용자가 슬랙 채널 내에서 봇을 멘션하여 문서를 기반으로 질문하더라도 빈 결과가 오거나 답변을 완성하지 못했습니다.

### 2.2 원인 분석
- 로컬 Ollama 모델 인프라의 내부 구동 방식에 따라 신규 임베딩 규격 엔드포인트 `/api/embed` (키값: `embeddings`)와 구형 규격 `/api/embeddings` (키값: `embedding`) 간의 JSON 응답 키 불일치 및 디코딩 예외가 존재했습니다.
- 로컬 GPU/CPU 자원 한계로 인한 임베딩 응답 연동 지연(Timeout) 시간 초과 현상이 발생했습니다.

### 2.3 해결 조치 및 고도화
1. **Ollama 다중 규격 통합 연동**:
   - `/api/embed` (최신)와 `/api/embeddings` (구형) 규격을 모두 순차적으로 검증하고, 반환되는 JSON 데이터 구조(`embeddings`와 `embedding` 리스트 검색)를 전부 소화하도록 파싱 로직을 정교화했습니다.
   - 연산 타임아웃을 안전하게 **10초**로 확대하여 대용량 단락 임베딩 중 발생하는 세션 락을 방지했습니다.
2. **Gemini API 기반 2중 임베딩 폴백(Fallback) 안전망 장착**:
   - 로컬 Ollama 서비스가 꺼져 있거나 로드 오류가 발생하는 극한의 상황을 대비하여, RAG 가동 시 `GEMINI_API_KEY`를 바탕으로 구글의 **`models/text-embedding-004` (768차원)** 모델을 사용하여 클라우드 임베딩으로 자동 전환하는 폴백 방어선을 장착했습니다. 이제 어떠한 로컬 오류 속에서도 챗봇 지식이 차질 없이 반영됩니다.
3. **LLM 타임아웃 락 해제**:
   - 긴 가이드라인 문맥을 정독하여 답변을 도출할 때 발생할 수 있는 소형 LLM의 타임아웃을 방지하기 위해 생성 타임아웃을 **90초**로 확장하여 긴 온보딩 질의응답 시의 안정성을 극대화했습니다.

## 3. 파일 업로드 버전 관리 및 히스토리 관리 고도화

### 3.1 요구 사항
기존에는 `2. 과업지시서.pdf`와 `신입사원 온보딩.pdf`라는 고정된 명칭으로 파일이 업로드되어 이전에 관리자가 올린 문서가 덮어씌워지며 과거 이력이 소실되는 문제가 존재했습니다. 새롭게 업로드되는 문서는 `.verxx` 넘버링을 통해 히스토리가 고유하게 보관되어야 합니다.

### 3.2 구현 세부 내용
1. **버전 파일명 탐색 엔진 탑재 (`get_versioned_filename`)**:
   - 백엔드에 파일 저장 전용 버전 감지 유틸리티를 추가했습니다.
   - 로컬 디렉터리를 스캔하여 동일명 파일이 존재할 경우 `.ver01`, `.ver02` 등의 접미사를 파일명 중간에 주입한 고유한 경로를 반환합니다.
     - 예: `2. 과업지시서.ver01.pdf` ➡️ `2. 과업지시서.ver02.pdf`
2. **파이프라인 전역 버전 동기화**:
   - 자동 생성된 버전 파일명은 아래 파이프라인 요소에 100% 동일하게 일치되어 흐릅니다:
     - **로컬 임시 캐시**: `c:\auto_onbording\` 하위에 실제 버전화되어 저장.
     - **구글 드라이브**: 클라우드 저장소 내에 동일한 버전화 이름으로 생성 백업 (히스토리 완벽 보존).
     - **RAG 지식 인덱스**: `vector_db.json` 내 `metadata.source` 정보로 버전명이 매핑되어 청킹이 저장되므로, 챗봇이 최신 및 구버전 이력을 모두 활용 가능.
     - **슬랙 채널 배포**: 신입사원이 사용하는 슬랙 채널 내에 버전명이 표시된 정교한 파일명으로 업로드되어 배포 완료.

# 온보딩 RAG 챗봇 구성

## 1. 개요
신입 근로자가 슬랙 채널 내에서 챗봇(`@Auto_Bot1`)을 호출하여 과업지시서 내용, 업무 절차, 온보딩 규칙 등에 대해 질문하면, 사전에 구축된 RAG(Retrieval-Augmented Generation) 파이프라인을 통해 정확도 높은 맞춤형 답변을 제공하는 AI 어시스턴트 기능입니다.

## 2. RAG 파이프라인 4단계 구조 (`rag_service.py`)
1. **데이터 추출(Data Extraction)**:
   - `PyMuPDF (fitz)` 라이브러리를 사용하여 구글 드라이브/슬랙 배포 파이프라인에서 수집된 PDF 문서(과업지시서, 온보딩 가이드 등)의 원시 텍스트를 추출합니다.
2. **청킹(Chunking)**:
   - `langchain.text_splitter.RecursiveCharacterTextSplitter`를 활용해 추출된 전체 텍스트 문서를 의미가 단절되지 않는 적절한 사이즈(예: `chunk_size=500`, `chunk_overlap=50` 등)의 블록으로 분할합니다.
3. **임베딩(Embedding)**:
   - 로컬 구동 오픈소스 모델인 `Ollama(nomic-embed-text)` 또는 `Gemini Embedding` 모델 API를 사용하여 각 텍스트 청크를 다차원 벡터 배열로 변환합니다. 변환된 벡터들은 `vector_db.json`과 같은 로컬 인메모리/파일 스토리지에 텍스트 원본과 매핑되어 실시간 색인(Indexing)됩니다.
4. **검색(Retrieval)**:
   - 슬랙으로부터 유저의 질의문 텍스트가 수신되면, 질의문 역시 동일한 방식으로 임베딩합니다. 이후 코사인 유사도(Cosine Similarity) 알고리즘을 사용해 질의 벡터와 가장 유사도가 높은(관련성이 깊은) 텍스트 청크 상위 K개(Top-K)를 지식베이스에서 추출하여 컨텍스트로 반환합니다.

## 3. 답변 생성 모델 연동 및 프롬프팅
- **LLM 엔진**: 구글의 `Gemini API (gemini-1.5-pro/flash)`를 메인 생성 모델로 활용합니다.
- **프롬프트 엔지니어링**: RAG를 통해 Retrieval 된 컨텍스트(사내 문서 내용)를 템플릿의 변수로 주입합니다. 시스템 지시문에는 "제공된 컨텍스트 기반으로만 사실적으로 답변할 것", "정중하고 전문적인 한국어 톤앤매너를 유지할 것" 등을 명시하여 할루시네이션(환각)을 최소화합니다.

## 4. 슬랙 비동기 이벤트 챗봇 처리 흐름
- 유저가 슬랙 내에서 `@Auto_Bot1 질문내용` 포맷으로 챗봇을 멘션하면 `main.py`의 `/slack/events` 엔드포인트가 웹훅 이벤트를 수신합니다.
- 슬랙 API는 웹훅에 대해 3초 내 응답(HTTP 200)을 요구하므로, API 응답은 즉시 반환하고 실제 무거운 RAG 검색 및 LLM 추론 로직은 `async_rag_chat_process` 함수를 통해 FastAPI 백그라운드 스레드에서 비동기 처리합니다.
- 추론이 완료되면 `slack_service`의 `send_message` 메서드를 통해 최초 유저가 질의한 원본 메시지의 `thread_ts` 속성을 지정, 해당 스레드(답글) 공간에 자연스럽게 챗봇의 답변을 이어 달게 됩니다.

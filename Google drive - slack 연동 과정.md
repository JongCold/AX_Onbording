# Google Drive - Slack 연동 과정 (2차 개발 고도화 반영)

## 1. 전체 연동 파이프라인 아키텍처
본 시스템은 대시보드 웹 인터페이스와 구글 클라우드 스토리지, 슬랙 메신저, 그리고 로컬 RAG 인프라를 연결하는 통합 온보딩 파이프라인을 탑재하고 있습니다.

```
[관리자 대시보드 폼 전송] 
       │ (이름, 이메일, 채널 선택, PDF 파일 2개 업로드)
       ▼
[FastAPI 백엔드 수신 (/onboard-web)]
       │ 
       ├─► [버전 넘버링 처리] -> 2. 과업지시서.ver01.pdf / 신입사원 온보딩.ver01.pdf 로컬 캐싱
       │
       ├─► [구글 드라이브 업로드] -> GoogleDriveService.upload_file() (클라우드 백업)
       │
       ├─► [RAG 인덱싱] -> rag_service.index_pdf() (버전별 텍스트 청킹/임베딩 업데이트)
       │
       └─► [슬랙 채널 자동 연동]
               │
               ├─► [채널 매핑] -> 기존 채널 지정 또는 새로운 전용 채널(ot-proj-...) 동적 개설
               ├─► [봇 조인 보증] -> 봇 자체 가입 실패 시 SLACK_USER_TOKEN을 사용해 봇 강제 초대
               ├─► [파일 배포] -> .verxx.pdf 버전별 가이드라인 슬랙 채널 직접 업로드
               └─► [웰컴 인사] -> Gemini AI 활용 맞춤형 온보딩 환영 메시지 및 스레드 챗봇 안내
```

## 2. 세부 연동 프로세스

### 2.1 파일 수신 및 버전 관리 (History Control)
- 관리자가 웹 대시보드에서 PDF 파일을 업로드하면, 백엔드에서 `get_versioned_filename` 함수를 통해 기존 파일들과의 중복 여부를 검사합니다.
- 중복 시 `.ver01.pdf`, `.ver02.pdf` 등 순차적으로 넘버링된 버전 접미사를 붙여 고유한 파일명으로 로컬 디렉터리에 복사 저장합니다.

### 2.2 구글 드라이브 백업 및 클라우드 업로드
- 로컬 스토리지에 복사 저장된 고유 버전의 파일명(`*.verxx.pdf`) 그대로 `GoogleDriveService.upload_file` 메서드를 호출하여 구글 드라이브의 원격 지정 폴더(`GOOGLE_DRIVE_FOLDER_ID`)로 업로드합니다.
- 이를 통해 클라우드에 온보딩 가이드 및 과업지시서의 버전별 역사(History)가 고스란히 영구 보관됩니다.

### 2.3 RAG 벡터 DB 동적 데이터 머징
- 업로드 완료된 버전별 PDF 문서는 `rag_service.index_pdf` 로직에 투입되어 실시간으로 텍스트를 청킹하고 임베딩합니다.
- 신규 버전 파일은 중복 덮어쓰기가 아니라 고유 소스로 벡터 DB(`vector_db.json`)에 병합 추가되므로, 챗봇은 최신 버전 문서를 학습하여 더욱 다채롭고 역사적인 지식을 바탕으로 질의응답을 수행합니다.

### 2.4 슬랙 문서 배포 및 웰컴 가이드
- 파이프라인의 최종 단계에서, 지정되었거나 새로 개설된 슬랙 채널로 로컬의 고유 버전 파일(`.verxx.pdf`)을 `slack_service.upload_file`로 업로드합니다.
- 채널에 성공적으로 진입한 신입 근로자는 채널 상단에 배포된 최신 버전의 파일을 다운로드하고, 챗봇 멘션 시 RAG 지식을 통해 실시간 답변 서비스를 제공받습니다.
